<objective>
Harden PII handling for the new identity domain. Extend the backend and frontend
sensitivity inferers and the PII log sanitizer so identity attributes
(name, address, dob, passport, national_id) are classified and masked before they
can reach a log, error reporter, or debug surface.

Purpose: Today `_infer_sensitivity` only flags ssn/tax/account_number/routing as
restricted and risk/portfolio/holdings/income as confidential; the sanitizer masks
only email and phone. Identity PII (name, address, dob) currently flows unmasked.
Sensitivity is a substring heuristic, so the rule must be defensive â€” including
classifying `ssn`/`social_security` as restricted EVEN THOUGH no SSN is ever stored
(D-A). SSN here is only a classification, never a stored field.

Output: extended backend + frontend sensitivity inferers, extended sanitizer with
tests, and passing governance gates.
</objective>
