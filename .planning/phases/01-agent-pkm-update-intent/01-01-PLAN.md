<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM slots → frontend | `field_path`, `proposed_value`, `current_value` arrive as unvalidated strings from the LLM tool call |
| `build` callback → PKM write | The field patch is applied to authoritative `context.currentDomainData` — LLM's `current_value` is display-only, not used in write |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | `applyFieldPatch` | mitigate | Use `structuredClone` to prevent mutation of original domain data; patch is applied to a fresh copy inside the `build` callback, not to cached state |
| T-01-02 | Tampering | `saveAgentPkmUpdate` `build` callback | mitigate | Write uses `context.currentDomainData` (from `prepareDomainWriteContext` — fresh optimistic-lock read), not the LLM-supplied `current_value` which may be stale |
| T-01-03 | Information Disclosure | `summary` payload | accept | `summary` contains `field_path` and `proposed_value` — these are already present in the stored PKM domain; no new disclosure surface |
| T-01-SC | Tampering | npm/pip installs | accept | No new packages installed in this plan; `applyFieldPatch` is hand-rolled per Risk 2 mitigation in RESEARCH.md |
</threat_model>
